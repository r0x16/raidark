# RDK-010 — Driver JetStream consumer con DLQ, backoff, clasificación de errores y modo broadcast.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/events/driver/JetStreamDomainEventsProvider`
- **Épica técnica:** `EP-RDK-EVENTS` — Bus de eventos de dominio

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Implementar la suscripción a JetStream sobre pull consumers durables, con política de reentrega, clasificación de errores, DLQ a un stream paralelo y un modo broadcast efímero per-instancia para cargas tipo fanout.
- **Cómo:**
  - Estructura `JetStreamConsumer` que implementa el método `Subscribe(handler EventListener)` del puerto existente.
  - **Modo durable (default)** — pull consumer durable por servicio/intención:
    - Nombre durable provisto por el consumidor del framework (Raidark valida formato no-vacío, sin imponer estructura semántica).
    - `max_deliver` configurable, default `5`.
    - Backoff exponencial con jitter, schedule por defecto `1s, 4s, 15s, 60s, 300s`, parametrizable.
    - `ack_wait` configurable.
    - Clasificación de errores:
      - `ErrTransient` o `error.Transient()` → `nak` (con backoff).
      - `ErrPermanent` o `error.Permanent()` → `ack` + envío a DLQ.
      - Sin clasificar → tratado como transitorio hasta `max_deliver`, luego DLQ.
    - DLQ vía stream paralelo (nombre por env, p. ej. `APP_EVENTS_DLQ`) con subject `dlq.{original_subject}` y metadatos en headers (`delivery_count`, `last_error`, `last_attempt_at`, `original_subject`).
  - **Modo broadcast** — durable efímero per-instancia con `DeliverPolicy=New`:
    - Cada instancia recibe todos los mensajes (no compete por queue group).
    - Útil para gateways que mantienen estado en memoria (por ejemplo, conexiones WebSocket) y necesitan reaccionar a eventos sin coordinarse entre réplicas.
  - API:
    ```go
    type SubscribeOptions struct {
        Mode         SubscribeMode // Durable | Broadcast
        DurableName  string         // requerido si Mode=Durable
        Subject      string
        MaxDeliver   int
        AckWait      time.Duration
        Backoff      []time.Duration
        DLQSubject   func(envelope EventEnvelope) string // default: "dlq." + subject original
    }

    type EventError interface {
        error
        Transient() bool
        Permanent() bool
    }

    var (
        ErrTransient = errors.New("transient")
        ErrPermanent = errors.New("permanent")
    )
    ```
  - Variables de entorno:
    ```
    NATS_DLQ_STREAM=APP_EVENTS_DLQ
    NATS_MAX_DELIVER_DEFAULT=5
    NATS_BACKOFF_SCHEDULE=1s,4s,15s,60s,300s
    NATS_ACK_WAIT_DEFAULT=30s
    ```
- **Por qué:** JetStream entrega at-least-once. Sin política explícita de reentrega, clasificación de errores y DLQ, los handlers buggy bloquean el consumer indefinidamente. Sin modo broadcast, los gateways realtime no pueden mantener estado per-instancia.
- **Cuándo:** Bloqueante: `RDK-008`, `RDK-009`. Recomendado: `RDK-003` para métricas de redeliveries y DLQ.

## Criterio de aceptación
- Driver implementa el puerto consumer del provider existente.
- Tests de integración con NATS embebido:
  - Mensaje con handler que retorna `ErrTransient` → reintenta y eventualmente acaba en DLQ tras `max_deliver`.
  - Mensaje con handler que retorna `ErrPermanent` → ack inmediato + DLQ.
  - Mensaje con handler exitoso → ack, no DLQ, métricas incrementadas.
  - Modo broadcast: dos consumers distintos reciben el mismo mensaje.
  - Headers W3C trace presentes en el contexto del handler.
- Métricas (con RDK-003 disponible): `events_consumed_total`, `events_redeliveries_total`, `event_processing_duration_ms` poblados.
- Documentado en `docs/events/jetstream-consumer.md` con ejemplos de cada modo y de clasificación de errores.

## Fuera de alcance
- Push consumers (sólo pull).
- Particionado por aggregate id en el broker (responsabilidad del subject mapper).
- Replay manual desde DLQ (es operación humana / herramienta aparte).

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que consume eventos de dominio.
- **Historia:** Como desarrollador que consume eventos, quiero un driver con política de reentrega, DLQ y dos modos de consumo (durable y broadcast), para no perder mensajes en errores transitorios, no bloquear el consumer en errores permanentes y soportar gateways realtime per-instancia sin coordinación extra.
- **Valor esperado:** Los handlers se concentran en lógica de dominio; reintentos, DLQ y modos los maneja el driver.
