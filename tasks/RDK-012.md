# RDK-012 — Dedup en consumer (`processed_events`).

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/events/dedup`
- **Épica técnica:** `EP-RDK-EVENTS` — Bus de eventos de dominio

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Garantizar que un mismo evento no produzca efectos duplicados en el consumer aunque JetStream lo entregue más de una vez (es at-least-once por diseño).
- **Cómo:**
  - Modelo GORM `ProcessedEvent`:
    | Columna | Tipo |
    |---|---|
    | `idempotency_key` | text PK |
    | `event_name` | text |
    | `consumer_name` | text |
    | `processed_at` | timestamptz |
    Índice secundario `(consumer_name, processed_at)` para limpieza posterior.
  - Wrapper sobre el handler que el driver consumer (RDK-010) invoca:
    ```go
    func WithDedup(consumerName string, db *gorm.DB, handler EventHandler) EventHandler
    ```
    Comportamiento por mensaje:
    1. Abrir tx.
    2. `SELECT ... FROM processed_events WHERE idempotency_key=? FOR UPDATE`.
    3. Si existe → commit, ack del mensaje, NO ejecuta handler.
    4. Si no existe → ejecutar handler dentro de la misma tx.
       - Éxito → INSERT en `processed_events` + commit + ack.
       - Error transitorio → rollback + nak.
       - Error permanente → rollback + ack + envío a DLQ (delegado al driver).
  - Cada servicio consumidor agrega `&ProcessedEvent{}` en su lista de modelos (vía el módulo de eventos que el consumidor del framework declare).
- **Por qué:** Sin dedup en consumer, un evento entregado dos veces produce efectos duplicados (notificaciones repetidas, contadores doblados, acciones idempotentes que sí son idempotentes pero acciones no idempotentes que rompen). La dedup nativa de JetStream protege al broker, no al handler.
- **Cuándo:** Bloqueante: `RDK-010`.

## Criterio de aceptación
- Modelo `ProcessedEvent` exportable como hook de migración.
- Wrapper `WithDedup` exportado.
- Tests con NATS embebido + Postgres testcontainer:
  - Mensaje entregado dos veces → handler ejecuta una sola vez.
  - Mensaje cuyo handler falla con `ErrTransient` → no se inserta en `processed_events`, reintenta.
  - Mensaje cuyo handler falla con `ErrPermanent` → no se inserta en `processed_events`, va a DLQ.
  - Race condition: dos workers compiten por el mismo `idempotency_key`. Uno gana (insert), el otro hace rollback y ack tras detectar la fila.
- Documentado en `docs/events/consumer-dedup.md` con ejemplo de wrapping de un handler.

## Fuera de alcance
- Cleanup automático de filas viejas (responsabilidad del operador o job aparte; recomendar TTL operativo en docs).
- Dedup global cross-consumer (cada `consumer_name` es independiente; eso es deliberado para que distintos consumers procesen el mismo evento).

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que consume eventos cuyas reacciones no son idempotentes.
- **Historia:** Como desarrollador que reacciona a eventos con efectos no idempotentes (otorgar EXP, enviar email, contar likes), quiero un wrapper de dedup transaccional, para que las redeliveries inevitables de JetStream no me dupliquen efectos.
- **Valor esperado:** El handler se concentra en lógica; la dedup es invisible y verificable por la tabla `processed_events`.
