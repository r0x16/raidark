# RDK-019 — Helper de particionado mensual nativo de Postgres.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/migration/partitioning`
- **Épica técnica:** `EP-RDK-DB` — Migraciones y persistencia

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Entregar helpers para gestionar tablas particionadas por `RANGE(created_at)` mensual en Postgres, ya que GORM no soporta particionado nativo de forma directa.
- **Cómo:**
  - API de migración:
    ```go
    func CreateMonthlyPartition(tx *gorm.DB, parentTable string, month time.Time) error
    func DetachMonthlyPartition(tx *gorm.DB, parentTable string, month time.Time) error
    func PartitionName(parentTable string, month time.Time) string  // p. ej. "events_2026_04"
    ```
    Implementa SQL:
    ```sql
    CREATE TABLE IF NOT EXISTS <child>
        PARTITION OF <parent>
        FOR VALUES FROM ('YYYY-MM-01') TO ('YYYY-MM+1-01');
    ```
  - Job de pre-creación opcional:
    ```go
    type PartitionAheadOptions struct {
        ParentTable   string
        MonthsAhead   int           // crear N meses por adelantado
        Interval      time.Duration // cada cuánto correr el chequeo
    }
    func StartPartitionAheadJob(db *gorm.DB, opts PartitionAheadOptions) (cancel func())
    ```
  - El parent table debe existir antes (creado por SQL custom de RDK-020 como `PARTITION BY RANGE (created_at)`).
  - Validación: `parentTable` validado como identificador SQL seguro (regex `^[a-z_][a-z0-9_]*$`); `month` se reduce a primer día UTC del mes.
- **Por qué:** Tablas que crecen rápido (mensajes de chat, logs de auditoría, eventos históricos) se manejan mejor con particionado nativo: queries por rango son más rápidos, archive/drop de meses viejos es instantáneo. GORM no provee herramientas; el consumidor lo necesita expuesto.
- **Cuándo:** Bloqueante recomendado: `RDK-020` (necesario para crear el parent table con `PARTITION BY` mediante SQL custom).

## Criterio de aceptación
- Tests con Postgres testcontainer:
  - `CreateMonthlyPartition` para mes M crea la tabla `parent_YYYY_MM` con bordes correctos.
  - Inserts con `created_at` dentro de M caen en la partición correspondiente.
  - Inserts fuera de cualquier partición declarada → error nativo de Postgres.
  - `DetachMonthlyPartition` separa la partición sin perder datos.
  - Job `StartPartitionAheadJob` con `MonthsAhead=2` crea M+1 y M+2 si faltan.
- Documentado en `docs/migration/postgres-partitioning.md` con ejemplo completo (creación de parent + uso del helper + job opcional).

## Fuera de alcance
- Particionado por hash o list (sólo RANGE mensual en v1).
- Archivado a almacenamiento frío (responsabilidad de infraestructura).
- Auto-drop de particiones viejas (decisión operativa, no de Raidark).

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que persiste tablas de alto volumen ordenadas en el tiempo.
- **Historia:** Como desarrollador que persiste eventos o mensajes con alto volumen mensual, quiero helpers para crear particiones mensuales y precrearlas con anticipación, para no quedarme sin partición a fin de mes y poder archivar meses viejos sin tocar índices del resto.
- **Valor esperado:** El servicio mantiene su propia tabla particionada con un job ligero que precrea N meses adelante; archivado y drop son operaciones de un comando.
