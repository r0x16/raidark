# RDK-021 — Drivers exportables y composables desde el exterior.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** Toda la superficie pública de `shared/`
- **Épica técnica:** `EP-RDK-API` — Superficie pública del framework

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Auditar y, donde haga falta, refactorizar la visibilidad de los drivers nuevos (RDK-005..RDK-020) para que un consumidor externo pueda construir provider factories propias que compongan Raidark con valores opinados.
- **Cómo:**
  - Para cada driver entregado en RDK-005..RDK-020, verificar:
    - Tipos públicos: structs, interfaces, opciones de configuración exportadas.
    - Constructores accesibles: `NewX(opts XOptions) *X` en lugar de `newX` privado.
    - Hooks de lifecycle (`Connect`, `Close`, `Start`, `Stop`) exportados donde apliquen.
    - Provider factories registrables por el consumidor sin tener que vivir dentro de `shared/`.
  - Documentar el patrón "factory opinada" en `docs/extending.md`:
    - Cómo un consumidor puede crear una factory propia que, en vez de leer env vars, reciba opciones tipadas y devuelva el provider configurado.
    - Cómo registrar esa factory en el provider hub de Raidark.
  - Asegurar que ninguna decisión de configuración esté oculta detrás de un singleton interno; todo provider debe poder existir en múltiples instancias en el mismo proceso (por ejemplo, dos clientes HTTP con baseURL distinto, dos `JetStreamPublisher` con streams distintos).
  - Confirmar que el provider hub permite registrar providers post-bootstrap (para escenarios donde un módulo registra el suyo después del setup principal).
- **Por qué:** Si los drivers viven detrás de tipos privados o factories no extensibles, un consumidor externo termina copiando código o monkey-patcheando para componer Raidark. La capacidad de envolver y opinar sobre los providers es lo que permite a un proyecto base (sobre Raidark) entregar configuraciones por defecto a los servicios que lo usan.
- **Cuándo:** Bloqueante: todas las RDK anteriores que entregan drivers (RDK-005..RDK-020).

## Criterio de aceptación
- Cada driver de RDK-005..RDK-020 tiene tipos y constructores públicos verificados.
- Existe `docs/extending.md` con un ejemplo end-to-end: un consumidor construye una `OpinatedOutboxFactory` que envuelve el dispatcher con poll interval y batch size predefinidos.
- Test ejemplo en `examples/external-factory/` que demuestra crear una factory externa al árbol `shared/` y registrarla en el provider hub.

## Fuera de alcance
- Plugin system con carga dinámica de `.so` (no necesario; basta con composición en tiempo de compilación).
- Versionado semántico estricto de la API pública (queda como decisión separada del proyecto Raidark).

## Historia de usuario relacionada
- **Actor:** Equipo que mantiene un proyecto base sobre Raidark y quiere distribuir defaults opinados a sus servicios.
- **Historia:** Como mantenedor de un proyecto que estandariza Raidark para varios servicios, quiero que todos los drivers sean exportables y composables, para escribir factories opinadas que envuelvan a Raidark sin tener que forkearlo.
- **Valor esperado:** Un proyecto consumidor puede entregar a sus servicios una capa fina sobre Raidark (`coreprov.OutboxDispatcherFactory{}`, `coreprov.ImagePipelineFactory{}`) sin tocar el repositorio base.
