# RDK-003 — Observabilidad base: logs estructurados, métricas Prometheus y W3C trace context.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/observability`
- **Épica técnica:** `EP-RDK-OBSERVABILITY` — Observabilidad transversal

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** In Progress
- **Quién:** DEV
- **Qué:** Entregar la observabilidad base que cualquier servicio construido sobre Raidark necesita: logs estructurados con correlación, métricas Prometheus mínimas y propagación W3C trace context entre HTTP y bus de eventos.
- **Cómo:**
  - Extender `LogProvider` actual con un adapter JSON sobre `slog` (Go 1.21+) o `zap`. Toggleable por env (`LOG_FORMAT=json|text`).
  - Helper `log.FromContext(ctx)` que devuelve un logger con campos automáticos: `trace_id`, `span_id`, `service`, y `event_id` cuando aplique.
  - Middleware Echo `observability.HTTPMetrics()` que expone `http_requests_total{status,endpoint}` y `http_request_duration_ms{endpoint}` (histograma con buckets razonables: 5, 25, 100, 500, 1000, 5000 ms).
  - Counter/gauge/histogram registrados para eventos: `events_published_total{subject,outcome}`, `events_consumed_total{subject,consumer,outcome}`, `events_redeliveries_total{subject,consumer}`, `event_processing_duration_ms{subject,consumer}`, `outbox_pending_gauge`.
  - Middleware `observability.W3CTrace()` que parsea `traceparent` / `tracestate` entrantes o genera uno nuevo, pueblan `trace_id` y `span_id` en `echo.Context`.
  - Provider factory `PrometheusMetricsProvider` que monta `/metrics` en el router cuando `METRICS_ENABLED=true`.
  - Helper para drivers NATS/HTTP de propagar `traceparent` en headers (consumido por RDK-009/010 y RDK-016).
- **Por qué:** Sin observabilidad mínima estandarizada, cada servicio reinventa logging y métricas con campos distintos, y debuggear cross-service en presencia de eventos asíncronos se vuelve impracticable.
- **Cuándo:** Bloqueante recomendado: `RDK-002` (para reusar `X-Correlation-ID` como `trace_id` cuando ya viene poblado).

## Variables de entorno
```
LOG_FORMAT=json
LOG_LEVEL=INFO
METRICS_ENABLED=true
METRICS_PATH=/metrics
SERVICE_NAME=<nombre del servicio>
```

## Criterio de aceptación
- Paquete `shared/observability` con los middlewares y los métricas registrados.
- Logger automático inyecta `trace_id`, `span_id`, `service`.
- `/metrics` expone counters/histograms con labels documentados.
- Tests:
  - Log emitido contiene `trace_id` cuando el contexto lo trae.
  - Histogram de HTTP recibe muestras tras una request real (test con Echo).
  - W3C: `traceparent` válido entrante se propaga; ausente genera uno nuevo.
- Documentado en `docs/observability/logging.md`, `docs/observability/metrics.md`, `docs/observability/tracing.md`.

## Fuera de alcance
- OpenTelemetry collector y exporters (OTLP). Se deja la propagación W3C lista para que el siguiente paso sea integrar OTel sin tocar handlers.
- Dashboards Grafana.
- Alerting.

## Historia de usuario relacionada
- **Actor:** SRE / on-call que opera servicios construidos sobre Raidark.
- **Historia:** Como SRE que opera servicios construidos sobre Raidark, quiero logs JSON con `trace_id` automático, métricas Prometheus en `/metrics` y propagación W3C trace, para correlacionar requests entre servicios y entre HTTP y eventos sin instrumentar cada servicio a mano.
- **Valor esperado:** Cada servicio nuevo expone observabilidad base sin código adicional. Una traza HTTP que dispara un evento sigue siendo correlacionable cuando el evento es consumido en otro proceso.

### Iteración 1 (respondida)

1. **¿`LOGGER_TYPE=observability` como opt-in está bien, o el observability-logger debería ser el default y dejar `stdout` sólo como fallback legado?**
   Contexto: hoy `LoggerProviderFactory` mantiene `stdout` (legacy `StdOutLogManager`) como default y agrega `observability` como nuevo valor. Eso evita romper servicios actuales pero implica que ningún servicio recibe `trace_id` automático en logs hasta que cambien la env var. La tarea pide "extender LogProvider con un adapter JSON sobre slog… toggleable por env (LOG_FORMAT=json|text)" — implementé el toggle de FORMAT, pero el toggle de PROVIDER quedó conservador.
   Opciones:
   - a) Mantener como está (opt-in vía `LOGGER_TYPE=observability`).
   - b) Hacer del observability-logger el default y dejar `LOGGER_TYPE=stdout-legacy` (o similar) como escape.
   - c) Reemplazar `StdOutLogManager` por el nuevo logger (no hay opción de volver al viejo) — el legacy queda muerto.
   **Respuesta:** Deben quedar ambos disponibles, no le pongas legacy al otro ya que aún es valido. Que el observability sea el por defecto.

2. **¿El nuevo logger debe aplicar `LogDataSanitizer` (redacción de campos sensibles + truncado de structs) como hace `StdOutLogManager`?**
   Contexto: el `StdOutLogManager` corre `LogDataSanitizer` para redactar `password`, `token`, `cookie`, etc., y truncar a 500 chars. Mi `Logger` nuevo no lo aplica (pasa los valores directo a `slog.Any`). Lo dejé así porque la tarea no lo menciona y mezclar sanitización con auto-correlación complica el path; pero hoy ese sanitizador es la única red de seguridad anti-leak que tenemos.
   Opciones:
   - a) Replicar `LogDataSanitizer` en el nuevo logger (paridad con `StdOutLogManager`).
   - b) Dejarlo como está y documentar que la sanitización es responsabilidad del call-site.
   - c) Mover `LogDataSanitizer` a `shared/observability/log` y compartirlo entre ambos loggers.
   **Respuesta:** c

3. **¿`MetricsProviderFactory` debe registrarse como `baseProvider` siempre, o sólo cuando `METRICS_ENABLED=true`?**
   Contexto: hoy queda registrado siempre, pero `enabled=false` hace que `MountScrapeEndpoint` sea no-op. Las llamadas `Record…` siguen incrementando counters en memoria (no se ven, pero gastan CPU/mem). La ventaja de registrar siempre es que el código de los servicios no necesita chequear `Exists` cuando llama `metrics.Record…`.
   Opciones:
   - a) Mantener como está (siempre registrado, scrape opcional).
   - b) Registrar sólo cuando `METRICS_ENABLED=true` y forzar a callers a chequear `Exists`.
   - c) Mantener registrado siempre pero con un noop-provider cuando `enabled=false` (para que no haya overhead).
   **Respuesta:** b (Debes añadirlo de todas maneras en los providers del main.go de ejemplo)

4. **¿`W3CTrace` debe escribir `traceparent` en la respuesta siempre, o sólo cuando vino en la request?**
   Contexto: hoy lo escribe siempre (con el `trace_id` resuelto, sea heredado o nuevo) para que clientes con `curl -i` puedan ver el ID sin acceso a logs. El estándar W3C no obliga ni prohíbe esto. La alternativa es hacerlo sólo cuando el cliente lo envió, manteniendo el header como "fluido" sin imponerle nada al cliente que no lo pidió.
   Opciones:
   - a) Siempre (como está).
   - b) Sólo si vino en la request.
   - c) Sólo cuando `LOG_LEVEL=DEBUG` (para no saturar respuestas en prod).
   **Respuesta:** a

5. **¿La promoción `X-Correlation-ID` → `trace_id` (cuando el correlation-id es UUIDv7 sin guiones = 32 hex) está bien, o el `trace_id` siempre debe ser independiente del correlation-id?**
   Contexto: la tarea menciona en su descripción "para reusar `X-Correlation-ID` como `trace_id` cuando ya viene poblado". Lo implementé estricto: stripa guiones, valida 32 hex lowercase no-zero. UUIDv7 sin guiones cumple. Pero esto acopla los dos namespaces de IDs: si en el futuro el correlation-id deja de ser UUIDv7, la promoción rompe silenciosamente. El comportamiento alternativo sería tratarlos como independientes y dejar el correlation-id sólo en el envelope REST.
   Opciones:
   - a) Mantener la promoción (como está).
   - b) Independizar: `trace_id` siempre se genera/hereda de `traceparent`, ignorando el correlation-id.
   - c) Promoción opcional vía env var (`TRACE_PROMOTE_CORRELATION_ID=true|false`, default true).
   **Respuesta:** Acá la verdad yo entendía uqe el correlation-id era el id requerido para hacer traza, por lo que no entiendo por qué la tarea lo menciona como opcional, ten en cuenta que la tarea RDK-002 fue creada por el mismo agente que creó RDK-003, por lo tanto deberían ser compatibles, no entiendo la confusión.
