# RDK-013 — Helpers de auth basados en permisos.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/auth/permissions`
- **Épica técnica:** `EP-RDK-AUTH` — Autenticación y autorización

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Entregar middlewares Echo que protejan rutas exigiendo permisos del usuario autenticado, asumiendo que un middleware de auth previo (provider `casdoor` para producción o `array` para tests, los dos providers que Raidark soporta hoy) ya pobló los `Claims` en el contexto.
- **Cómo:**
  - API:
    ```go
    func RequirePermission(perm string) echo.MiddlewareFunc
    func RequireAny(perms ...string) echo.MiddlewareFunc
    func RequireAll(perms ...string) echo.MiddlewareFunc
    ```
  - Lectura de permisos desde el contexto:
    - Extractor por default para el provider `casdoor`:
      - Si el JWT trae un claim `permissions` (slice de strings), se usa directo.
      - Si el JWT trae roles, Raidark consulta el endpoint de permisos de Casdoor o un mapa rol→permisos provisto por configuración del consumidor (cualquiera sea el camino, queda documentado en `docs/auth/casdoor-permissions.md`).
    - Extractor por default para el provider `array` (mock de tests): lee permisos directamente del usuario mockeado.
    - El consumidor puede sobrescribir el extractor:
      ```go
      type PermissionExtractor func(c echo.Context) ([]string, error)
      func RegisterExtractor(e PermissionExtractor)
      ```
  - Comportamiento:
    - Sin claim de auth → 401 con envelope estándar (RDK-002), código `auth.unauthenticated`.
    - Claim presente pero sin permiso → 403, código `auth.forbidden`.
    - Permiso presente → next.
  - Logging:
    - 401 / 403 emiten log estructurado con `user_id` (si está), `endpoint`, `required_perm(s)`, `trace_id`.
- **Por qué:** Si cada handler implementa su propia verificación de permisos, los chequeos divergen y son fáciles de olvidar. Un middleware declarativo en la registración de la ruta hace explícito y auditable qué permiso protege cada endpoint.
- **Cuándo:** Bloqueante recomendado: `RDK-002` (envelope de error compartido).

## Criterio de aceptación
- Tests:
  - Endpoint con `RequirePermission("foo.bar")` y request sin claims → 401 con código `auth.unauthenticated`.
  - Mismo endpoint con claim que no incluye `foo.bar` → 403 con código `auth.forbidden`.
  - Mismo endpoint con claim que incluye `foo.bar` → 200.
  - `RequireAny`: pasa con cualquiera de los permisos.
  - `RequireAll`: pasa sólo con todos.
  - Extractor custom: registrar uno y verificar que se invoca.
- Documentado en `docs/auth/permissions.md` con ejemplo de uso y de registro de un `PermissionExtractor`.

## Fuera de alcance
- Catálogo de permisos: Raidark no impone nombres; cada consumidor define el suyo.
- Mapeo rol → permisos para providers distintos a `casdoor` y `array` (queda para cuando Raidark soporte un nuevo IDP).
- ABAC / reglas sobre recursos específicos (queda como mejora futura).

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que protege endpoints HTTP por permisos.
- **Historia:** Como desarrollador que registra rutas en Echo, quiero middlewares declarativos para exigir permisos, para que la protección viva al lado de la ruta y el chequeo no dependa de que recuerde llamar una función dentro del handler.
- **Valor esperado:** La protección de cada endpoint es leíble en una sola línea junto a su declaración. 401 vs 403 son consistentes en todo el servicio.
