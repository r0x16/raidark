# RDK-002 — Convenciones REST: envelope de error y paginación keyset.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/api/rest`
- **Épica técnica:** `EP-RDK-HTTP` — Capa HTTP base de Raidark

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** In Progress
- **Quién:** DEV
- **Qué:** Estandarizar la forma de los errores HTTP y la paginación que exponen los servicios construidos sobre Raidark.
- **Cómo:**
  - Tipo `RESTError` con campos `code` (string namespaced, p. ej. `forum.topic.not_found`), `message` (humano), `details` (mapa opcional con info de validación), `trace_id` (propagado del contexto).
  - Helper `rest.RenderError(c echo.Context, status int, e *RESTError) error` que serializa el envelope con shape estable.
  - Sentinels tipados: `ErrNotFound`, `ErrConflict`, `ErrForbidden`, `ErrValidation`, `ErrTransient`, `ErrPermanent`.
  - Mapper `rest.MapError(err error) (status int, restErr *RESTError)` que traduce sentinels a códigos HTTP estándar (400/401/403/404/409/413/422/429/500/502/503).
  - Tipo genérico `rest.Page[T any] { Items []T; NextCursor string; Limit int }` con utilities para encodear/decodear cursores opacos (base64 JSON con campos de orden).
  - Middleware `rest.CorrelationID()` que lee `X-Correlation-ID` (o genera uno UUIDv7 si no viene), lo inyecta en `echo.Context` y en el contexto de logs/traces.
- **Por qué:** Cada servicio construido a mano sobre Echo decide su propia forma de errores y paginación. Sin convenciones, los clientes tienen que parsear N envelopes distintos y debuggear cross-service es costoso.
- **Cuándo:** Sin bloqueantes (puede usar `RDK-001` para generar trace IDs si está disponible, pero no es estricto).

## Forma del envelope de error
```json
{
  "error": {
    "code": "<dominio>.<motivo>",
    "message": "Human readable",
    "details": { "field": "..." },
    "trace_id": "01J..."
  }
}
```

## Forma de la página
```json
{
  "items": [],
  "pagination": {
    "next_cursor": "opaque",
    "limit": 20
  }
}
```

## Criterio de aceptación
- Paquete `shared/api/rest` con envelope, helpers de paginación y mapper de errores.
- Sentinels exportados.
- Middleware `CorrelationID` instalable como `e.Use(rest.CorrelationID())`.
- Tests unitarios:
  - Round-trip de cursor opaco.
  - Mapper de errores para cada sentinel.
  - Render de envelope con y sin `details`.
  - Middleware: header presente se respeta, header ausente se genera.
- Documentado en `docs/rest/error-envelope.md`, `docs/rest/pagination.md`, `docs/rest/correlation-id.md`.

## Fuera de alcance
- OpenAPI / generación de specs.
- Negociación de contenido (siempre JSON en v1).

## Historia de usuario relacionada
- **Actor:** Cliente HTTP de un servicio construido sobre Raidark.
- **Historia:** Como cliente que consume servicios construidos sobre Raidark, quiero un envelope de error y una forma de paginación uniformes, para escribir un único parser y para correlacionar trazas usando el `trace_id` que me devuelve el servidor.
- **Valor esperado:** Los clientes (BFFs, otros servicios, dashboards) tratan errores y páginas con código compartido. La trazabilidad cross-service se obtiene gratis vía `X-Correlation-ID`.
