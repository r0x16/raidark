# RDK-005 — Storage adapter abstracto con driver filesystem.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/storage`
- **Épica técnica:** `EP-RDK-STORAGE` — Almacenamiento de objetos

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Completed
- **Quién:** DEV
- **Qué:** Definir una abstracción `StorageProvider` para almacenar objetos binarios (imágenes, PDFs, otros archivos) con visibilidad pública o privada y URLs firmadas, y entregar un driver `filesystem` v1 con HMAC para signed URLs.
- **Cómo:**
  - Interfaz Go:
    ```go
    type StorageProvider interface {
        Put(ctx context.Context, key string, r io.Reader, opts PutOptions) (PutResult, error)
        Get(ctx context.Context, key string) (io.ReadCloser, ObjectInfo, error)
        Delete(ctx context.Context, key string) error
        SignedURL(ctx context.Context, key string, ttl time.Duration) (string, error)
        PublicURL(key string) string
        Exists(ctx context.Context, key string) (bool, error)
    }
    type Visibility int
    const (
        VisibilityPublic Visibility = iota
        VisibilityPrivate
    )
    type PutOptions struct {
        Visibility  Visibility
        ContentType string
        Size        int64
    }
    type PutResult struct {
        Key       string
        SizeBytes int64
        ETag      string
    }
    type ObjectInfo struct {
        Key         string
        SizeBytes   int64
        ContentType string
        ModifiedAt  time.Time
    }
    ```
  - Driver `filesystem`:
    - Dos raíces: `STORAGE_PUBLIC_ROOT` y `STORAGE_PRIVATE_ROOT`.
    - `Put` recibe `io.Reader` y escribe streaming (sin buffer completo en memoria).
    - `SignedURL` para objetos `Private` se emula con HMAC firmado sobre `(key, expires_at)`; el handler estático interno expuesto por Raidark valida firma y TTL antes de servir.
    - `PublicURL` concatena `STORAGE_PUBLIC_BASE_URL + key`.
  - Convención obligatoria de keys (validada en helper, no impuesta en interfaz): `{namespace}/{usage}/{año}/{mes}/{uuid}{ext}`. El consumidor decide `namespace` y `usage`.
  - Provider factory `StorageProviderFactory` registrable en el provider hub.
- **Por qué:** Sin abstracción, cada servicio guarda archivos en su propia carpeta con nombres impredecibles. Una interfaz común permite migrar a S3/MinIO/GCS más adelante sin cambiar código de aplicación.
- **Cuándo:** Bloqueante recomendado: `RDK-001` (UUIDv7 para keys).

## Variables de entorno
```
STORAGE_DRIVER=filesystem
STORAGE_PUBLIC_ROOT=/storage/public
STORAGE_PRIVATE_ROOT=/storage/private
STORAGE_PUBLIC_BASE_URL=https://cdn.example.com
STORAGE_SIGNING_SECRET=<hex>
STORAGE_SIGNED_URL_DEFAULT_TTL=600s
```

## Criterio de aceptación
- Paquete `shared/storage` con interfaz, tipos y driver `filesystem`.
- Handler interno para servir objetos privados con verificación HMAC y TTL.
- Tests:
  - Round-trip Put/Get con `io.Reader` que no carga todo en memoria (test con tamaño > 10MB).
  - `SignedURL` válida dentro de TTL, expira cuando vence.
  - `SignedURL` rechazada si el HMAC no coincide.
  - `Delete` borra archivo y `Exists` devuelve false.
- Documentado en `docs/storage/adapter.md` y `docs/storage/filesystem-driver.md`.

## Fuera de alcance
- Driver S3 / MinIO / GCS (queda para tarea futura siguiendo la misma interfaz).
- Replicación / backups.
- Quotas por consumidor.

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que necesita persistir archivos binarios en un servicio.
- **Historia:** Como desarrollador construyendo un servicio sobre Raidark, quiero una interfaz de storage estándar con visibilidad pública/privada y URLs firmadas, para no acoplarme al filesystem desde el día uno y poder migrar a S3 cambiando solo configuración.
- **Valor esperado:** El código de aplicación habla con `StorageProvider`; cambiar de driver es un cambio de provider en bootstrap.

### 2026-05-02 — correcciones iteración 1

Se aplicaron las seis correcciones de la encuesta:

1. **`EchoStorageModule` retirado de `raidark.go`** — ahora es opt-in. Los servicios lo agregan explícitamente en su slice de `Run(modules)`. `docs/storage/adapter.md` actualizado con el ejemplo de registro correcto: tanto `StorageProviderFactory` (provider) como `EchoStorageModule` (módulo) son pasos opt-in.

2. **Adoptado `afero.BasePathFs` para operaciones de filesystem** — `afero v1.15.0` ya era dependencia indirecta; `go mod tidy` lo promovió a directa. `FilesystemStorageProvider` ahora tiene dos campos `afero.Fs` (`publicFs`, `privateFs`) en lugar de rutas raw. Todas las llamadas a `os.*` reemplazadas con equivalentes afero. Beneficios: protección path traversal por construcción (a nivel librería, no manual), y `afero.MemMapFs` puede inyectarse en tests sin tocar el FS real.

3. **Mejorado el check de path traversal en `ValidateKey`** — reemplazado `strings.Contains(key, "..")` (que daba falsos positivos en `a..b`) con verificación por segmento que rechaza solo `"."` y `".."` como segmentos aislados. `afero.BasePathFs` provee una segunda capa de protección independiente en cada llamada al filesystem.

4. **Extraído struct `keySegments` + `parseKeySegments()`** en `KeyHelper.go` — eliminado el desempaquetado posicional de cinco elementos. Campos con nombre (`namespace`, `usage`, `yearStr`, `monthStr`, `uuidExt`) hacen el código de validación autodocumentado.

5. **Documentada la rotación del signing secret en `docs/storage/filesystem-driver.md`** — nueva sección "Signing Secret Rotation" explica: los archivos NO están cifrados con el secreto (bytes crudos siempre accesibles vía `Get`), solo se rompe la verificación de signed URLs, procedimiento de rotación paso a paso.

6. **Reducida la complejidad cognitiva de `NewSignedUrlHandler`** — extraídos tres helpers en el mismo paquete `driver`: `parseAndValidateExpiry`, `verifyHMAC`, `servePrivateFile`, `signedURLForbidden`. El handler principal quedó en cuatro líneas planas de guards + dispatch.

### 2026-05-02 — verificación iteración 2

El usuario señaló una confusión conceptual entre `StorageProvider` (almacenamiento de objetos binarios) y `DatastoreProvider` (datos estructurados / base de datos). Auditoría completa de todos los archivos relacionados con storage confirmó:

- `shared/storage/domain/` — sin imports de base de datos, sin GORM, solo stdlib + `shared/ids`
- `shared/storage/driver/` — usa `afero` para operaciones de filesystem; sin imports de datastore
- `shared/providers/driver/StorageProviderFactory.go` — sin referencias a datastore
- `shared/api/driver/modules/EchoStorageModule.go` — sin referencias a datastore
- `docs/storage/adapter.md` — sin `DatastoreProviderFactory` en los ejemplos
- `docs/storage/filesystem-driver.md` — sin contenido de base de datos

**Causa raíz de la confusión:** El ejemplo de `adapter.md` de la sesión 1 incluyó por error `&driverprovider.DatastoreProviderFactory{}` junto a `StorageProviderFactory`, como si el almacenamiento de archivos dependiera de la base de datos. Esto fue corregido en la iteración 1 cuando `EchoStorageModule` se hizo opt-in y el ejemplo de registro fue reescrito.

**Sin cambios de código en esta iteración.** La implementación ya estaba limpia; solo fue necesaria la aclaración de documentación.
