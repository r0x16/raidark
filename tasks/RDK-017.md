# RDK-017 — Soporte WebSocket sobre Echo.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/ws`
- **Épica técnica:** `EP-RDK-HTTP` — Capa HTTP base de Raidark

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Entregar helpers para abrir conexiones WebSocket sobre Echo, con upgrade autenticado, ping/pong server-side automático y abstracción mínima sobre la librería subyacente.
- **Cómo:**
  - API:
    ```go
    type WSOptions struct {
        ReadTimeout  time.Duration
        WriteTimeout time.Duration
        PingInterval time.Duration
        MaxMessageBytes int64
        Subprotocols []string
    }
    func Upgrade(c echo.Context, opts WSOptions) (*WSConn, error)

    type Frame struct {
        Type    FrameType // Text | Binary
        Payload []byte
    }
    type WSConn interface {
        Read(ctx context.Context) (Frame, error)
        Write(ctx context.Context, f Frame) error
        Close(code CloseCode, reason string) error
        UserContext() context.Context
    }
    ```
  - Implementación recomendada: `nhooyr.io/websocket` (preferida; API más limpia, soporte nativo de `context.Context`). Alternativa `gorilla/websocket` documentada como driver alternativo.
  - Auth: el upgrade respeta cualquier middleware de auth previo. Si el contexto de Echo no trae claims, `Upgrade` falla con 401 (envelope estándar de RDK-002 es válido en la respuesta del handshake).
  - Ping/pong server-side automático configurable por `PingInterval`. La conexión se cierra si no responde dos pings seguidos.
  - Manejo del context cancellation: cerrar `UserContext()` o el contexto del request termina la conexión limpiamente con código `1001 Going Away`.
- **Por qué:** Los servicios realtime (chat, notificaciones push, presence) necesitan WebSocket. Sin un helper estándar, cada servicio reescribe el upgrade y el manejo de ping/pong, con bugs sutiles de cierre y leaks de goroutines.
- **Cuándo:** Sin bloqueantes estrictos. Recomendado tener `RDK-013` disponible si se quiere proteger el upgrade con permisos.

## Criterio de aceptación
- Tests:
  - Upgrade exitoso desde un test client.
  - Echo bidireccional de frames text y binary.
  - Ping/pong: si el cliente no responde, la conexión se cierra tras dos intervalos.
  - Cancelación del contexto cierra la conexión con código `1001`.
  - `MaxMessageBytes` excedido → conexión cerrada con código `1009 Message Too Big`.
- Documentado en `docs/ws/upgrade.md` con ejemplo completo de un endpoint WebSocket.

## Fuera de alcance
- Pub/sub interno entre conexiones (responsabilidad del consumidor; típicamente combinado con un consumer NATS broadcast).
- Backpressure por canal por usuario (el consumidor decide su política).
- Compresión `permessage-deflate` (queda como mejora futura).

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que construye un servicio realtime sobre Raidark.
- **Historia:** Como desarrollador construyendo un servicio realtime, quiero un helper de upgrade WebSocket sobre Echo con ping/pong y cierre limpio, para no manejar el handshake y los timeouts a mano y para que la conexión muera limpiamente cuando el contexto cancela.
- **Valor esperado:** Un endpoint WebSocket se escribe en pocas líneas; la conexión es robusta ante red flap y procesos que se reinician.
