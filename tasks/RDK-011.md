# RDK-011 — Outbox transaccional como librería reutilizable.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/events/outbox`
- **Épica técnica:** `EP-RDK-EVENTS` — Bus de eventos de dominio

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Implementar el patrón outbox como librería reutilizable, para que cualquier servicio pueda persistir un cambio de estado y publicar el evento asociado de forma **atómica**.
- **Cómo:**
  - Modelo GORM `OutboxEvent`:
    | Columna | Tipo | Notas |
    |---|---|---|
    | `id` | uuidv7 | PK |
    | `aggregate_type` | text | |
    | `aggregate_id` | text | |
    | `event_name` | text | |
    | `event_version` | int | |
    | `envelope` | jsonb | Envelope completo de RDK-008 serializado |
    | `status` | text | `pending`, `published`, `failed` |
    | `attempts` | int | |
    | `last_error` | text | |
    | `created_at` | timestamptz | |
    | `published_at` | timestamptz null | |
    Índice parcial sobre `status = 'pending'` para el dispatcher.
  - Helper:
    ```go
    func Enqueue(tx *gorm.DB, event DomainEvent, producer string, ctx context.Context) error
    ```
    Serializa el envelope (RDK-008) y lo inserta en `outbox_events` con `status='pending'` **dentro de la misma transacción** que pasó el caller.
  - Dispatcher background:
    - Goroutine arrancada por el bootstrap del servicio.
    - Polling configurable (`OUTBOX_POLL_INTERVAL_MS`, default 500).
    - Lee `outbox_events WHERE status='pending' ORDER BY created_at LIMIT N` (N configurable, default 100).
    - Publica vía `JetStreamPublisher` (RDK-009).
    - Éxito: marca `status='published', published_at=now()`.
    - Fallo: incrementa `attempts`, persiste `last_error`. Si `attempts >= MAX_OUTBOX_ATTEMPTS` (default 20), marca `status='failed'` y emite log con nivel ERROR.
    - Entrega ordenada por `aggregate_id` sin bloquear otros aggregates (advisory locks o particionamiento por hash de `aggregate_id`).
  - Provider factory `OutboxDispatcherFactory` registrable.
- **Por qué:** Sin outbox este escenario es inevitable: (1) servicio commitea cambio en DB, (2) llama a publish, (3) NATS / red / proceso cae, (4) estado quedó persistido pero el evento jamás se emitió. El outbox convierte el publish en "leer de una tabla y publicar", que es reintentable sin pérdida.
- **Cuándo:** Bloqueante: `RDK-008`, `RDK-009`.

## Variables de entorno
```
OUTBOX_POLL_INTERVAL_MS=500
OUTBOX_BATCH_SIZE=100
OUTBOX_MAX_ATTEMPTS=20
OUTBOX_AGGREGATE_PARTITIONS=8
```

## Criterio de aceptación
- Migración GORM del modelo `OutboxEvent` exportable como hook de migración (consumida por el módulo del servicio cliente).
- Test que valida atomicidad: si la transacción de negocio rollea, no queda fila en outbox.
- Test del dispatcher:
  - Fila `pending` termina `published` tras publish exitoso.
  - Fila con publish que falla queda `pending` con `attempts++`.
  - Fila que supera `MAX_OUTBOX_ATTEMPTS` queda `failed` y se loggea.
  - Dos eventos del mismo `aggregate_id` se publican en orden. Eventos de distintos aggregates no se bloquean entre sí.
- Documentado en `docs/events/outbox.md` con ejemplo completo: caso de uso que abre tx, persiste entidad, llama `Enqueue`, commitea.

## Fuera de alcance
- Outbox con CDC (Debezium).
- Cleanup automático de filas `published` antiguas (responsabilidad del operador o de un job aparte).
- Outbox sobre bases distintas de Postgres en v1 (el modelo GORM debe ser portable, pero las features avanzadas como particionamiento se prueban primero en Postgres).

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que necesita publicar eventos derivados de cambios de estado.
- **Historia:** Como desarrollador que cambia estado y publica eventos, quiero un outbox transaccional reutilizable, para no perder eventos cuando NATS o la red fallan justo entre el commit DB y el publish.
- **Valor esperado:** El caso de uso se reduce a "abrir tx → mutar → `Enqueue` → commit". El resto lo maneja el dispatcher.
