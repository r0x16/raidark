# RDK-020 — Soporte para migraciones SQL custom además de auto-migrate GORM.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/migration`
- **Épica técnica:** `EP-RDK-DB` — Migraciones y persistencia

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Permitir que un módulo declare migraciones SQL custom (extensiones, índices funcionales, constraints CHECK, particiones, triggers) además del auto-migrate de modelos GORM, y trackearlas para no re-ejecutarlas.
- **Cómo:**
  - Extender el `ApiModule` siguiendo la convención existente: como hoy un módulo que no tiene migraciones/seeds hereda un método vacío desde el embed `GenericApiModule` (o equivalente), aquí se agrega un método nuevo del mismo modo, de forma que **ningún módulo existente necesite cambios**:
    ```go
    type ApiModule interface {
        // ... métodos existentes
        GetCustomMigrations() []SQLMigration
    }

    // En el módulo base genérico (mismo lugar donde hoy viven los métodos
    // vacíos por default, p. ej. GetSeedData() devolviendo nil):
    func (m *GenericApiModule) GetCustomMigrations() []SQLMigration {
        return nil
    }

    type SQLMigration struct {
        ID   string  // identificador único, p. ej. "20260401_enable_pg_trgm"
        Up   string  // SQL idempotente preferido (CREATE EXTENSION IF NOT EXISTS, etc.)
        Down string  // opcional; vacío implica no-rollback
    }
    ```
    Cualquier módulo que ya embebe el módulo base genérico hereda el método vacío. Sólo los módulos que necesitan SQL custom lo sobrescriben.
  - Tabla `schema_migrations` (creada por Raidark al primer `dbmigrate`):
    | Columna | Tipo |
    |---|---|
    | `id` | text PK |
    | `module` | text |
    | `applied_at` | timestamptz |
    | `checksum` | text (sha256 del `Up`) |
  - Comportamiento de `dbmigrate`:
    1. Auto-migrate de modelos GORM (comportamiento actual, conservar).
    2. Para cada módulo, ejecutar sus `GetCustomMigrations()` ordenadas por `ID` (orden lexicográfico).
    3. Saltar las que ya estén en `schema_migrations` con el mismo `checksum`.
    4. Si una migración ya aplicada cambió de checksum → fallar con mensaje claro y código de salida no-cero.
  - CLI agregado: `go run ./main dbmigrate status` que lista migraciones aplicadas y pendientes por módulo.
- **Por qué:** El auto-migrate de GORM no maneja extensiones (`pg_trgm`, `unaccent`), índices funcionales (`LOWER(...)`), constraints CHECK, particiones, ni triggers. Servicios reales necesitan estos tres bloques además del schema GORM.
- **Cuándo:** Sin bloqueantes estrictos (es extensión de la migración existente).

## Criterio de aceptación
- Interfaz `ApiModule` extendida sin romper consumidores actuales (default `nil` o slice vacío para `GetCustomMigrations`).
- Tabla `schema_migrations` creada automáticamente al primer run.
- Tests:
  - Migración nueva se aplica una sola vez y queda registrada.
  - Migración con checksum cambiado falla con error explícito.
  - Migración cuyo SQL falla → no se registra y `dbmigrate` retorna no-cero.
  - `dbmigrate status` lista correctamente.
- Documentado en `docs/migration/custom-sql.md` con ejemplos: extensión Postgres, índice funcional, constraint CHECK, trigger, partición.

## Fuera de alcance
- Rollback automático en producción (queda como herramienta manual; la columna `Down` queda preparada para un futuro `dbmigrate down`).
- Migraciones expresadas en archivos `.sql` separados (en v1 se declaran como strings dentro del módulo Go).

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que necesita extensiones, índices funcionales, triggers o particiones en Postgres.
- **Historia:** Como desarrollador que necesita SQL custom además del auto-migrate de GORM, quiero declarar migraciones SQL en mi módulo y que `dbmigrate` las aplique idempotentemente, para no tener que correr scripts SQL a mano por servicio y por entorno.
- **Valor esperado:** Toda la evolución de schema (modelos + SQL custom) sale del repo del servicio y se aplica con un único comando.
