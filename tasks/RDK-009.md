# RDK-009 — Driver JetStream publisher.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/events/driver/JetStreamDomainEventsProvider`
- **Épica técnica:** `EP-RDK-EVENTS` — Bus de eventos de dominio

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Implementar un driver de publicación que cumpla el puerto `DomainEventsProvider` ya existente y publique eventos a NATS + JetStream con dedup nativo, propagación de trace y backpressure controlada.
- **Cómo:**
  - Estructura `JetStreamPublisher` que implementa el método `Publish(event DomainEvent) error` del puerto existente.
  - Dentro de `Publish`:
    - Envuelve el evento con `EventEnvelope` (RDK-008).
    - `Msg-Id` header NATS = `envelope.event_id` para dedup nativo de JetStream.
    - Headers NATS adicionales: `trace_id`, `span_id`, `producer`, `event_version`, `traceparent` W3C completo.
    - Subject viene del consumidor del framework (no es responsabilidad del driver inventar naming). El consumidor lo provee al construir el provider o por mapeo `event_name -> subject`.
    - `PublishAsync` con `PubAck` esperado bajo timeout configurable.
    - Backpressure: si el canal interno se llena, retorna `ErrTransient`.
  - Lifecycle:
    - `Connect()` al bootstrap.
    - `Close()` hace flush de mensajes pendientes con timeout máximo configurable.
  - Variables de entorno:
    ```
    NATS_URL=nats://nats:4222
    NATS_STREAM_NAME=<configurable, p. ej. APP_EVENTS>
    NATS_PUBLISH_TIMEOUT=5s
    NATS_PUBLISH_MAX_PENDING=1024
    NATS_RECONNECT_WAIT=2s
    NATS_MAX_RECONNECT=-1
    ```
  - Mapeo `event_name -> subject` configurable por consumidor:
    ```go
    type SubjectMapper func(envelope EventEnvelope) string
    ```
  - Provider factory `JetStreamPublisherFactory` registrable en el provider hub.
- **Por qué:** El `InMemoryDomainEventsProvider` actual es útil para tests pero no sobrevive a reinicios ni cruza procesos. Para servicios productivos hace falta un publisher persistente con dedup nativo.
- **Cuándo:** Bloqueante: `RDK-008`. Recomendado: `RDK-003` para inyectar trace headers automáticamente.

## Criterio de aceptación
- Driver implementa el puerto `DomainEventsProvider` actual sin breaking changes en consumidores que usen `InMemoryDomainEventsProvider`.
- Tests de integración con NATS embebido (test container o `nats-server` embedded):
  - Publicación exitosa entrega `PubAck` y el mensaje aparece en el stream con header `Nats-Msg-Id` = `event_id`.
  - Reenvío del mismo `event_id` no duplica el mensaje en el stream (dedup nativo verificado).
  - Headers `trace_id`, `span_id`, `producer`, `event_version`, `traceparent` presentes en el mensaje recibido.
  - Timeout de publish supera el límite → `ErrTransient`.
- Documentado en `docs/events/jetstream-publisher.md` con ejemplos de configuración y `SubjectMapper`.

## Fuera de alcance
- Outbox (es responsabilidad de `RDK-011`).
- Subscribe / consumer (RDK-010).

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que emite eventos de dominio desde un servicio.
- **Historia:** Como desarrollador que emite eventos de dominio, quiero un driver JetStream que publique con dedup nativo, propagación de trace y backpressure controlada, para que mis eventos lleguen exactamente una vez al broker y para correlacionarlos con la request HTTP que los originó.
- **Valor esperado:** Cambiar `DOMAIN_EVENT_PROVIDER_TYPE=jetstream` activa publicación productiva sin tocar código de aplicación.
