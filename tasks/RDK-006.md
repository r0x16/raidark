# RDK-006 — Pipeline de imágenes seguras (validación, re-encode, strip EXIF, variantes).

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/imaging`
- **Épica técnica:** `EP-RDK-MEDIA` — Procesamiento de medios

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Entregar un pipeline reutilizable que reciba una imagen subida por usuario, la valide, la re-encode, le quite metadata y genere variantes responsive antes de persistirla vía el storage adapter.
- **Cómo:**
  - Pipeline:
    1. **Validación:** magic bytes (jpeg/png/webp), dimensiones mínimas y máximas, tamaño máximo en bytes.
    2. **Re-encode:** salida WebP (calidad configurable) y JPEG fallback.
    3. **Strip metadata:** EXIF, IPTC y XMP completos. Sin excepciones.
    4. **Variantes:** generar `thumb`, `card`, `full` con dimensiones configurables por *usage*.
    5. **Persistencia:** vía `StorageProvider` (RDK-005) con keys `{namespace}/{usage}/{año}/{mes}/{uuid}_{variant}.{ext}`.
  - Configuración por *usage* declarable por el consumidor:
    ```go
    type ImageUsage struct {
        Name        string
        MaxBytes    int64
        MinWidth    int
        MaxWidth    int
        MinHeight   int
        MaxHeight   int
        Variants    []VariantSpec
    }
    type VariantSpec struct {
        Name      string  // thumb|card|full|...
        MaxWidth  int
        MaxHeight int
        Quality   int     // 1-100 webp/jpeg
    }
    ```
  - API:
    ```go
    func (p *ImagePipeline) Process(ctx context.Context, in io.Reader, usage string) (ImagePipelineResult, error)
    type ImagePipelineResult struct {
        BaseKey    string
        HashSha256 string
        Variants   []ImageVariant
    }
    type ImageVariant struct {
        Variant     string
        Format      string
        Key         string
        Width       int
        Height      int
        SizeBytes   int64
        ContentType string
    }
    ```
  - Dependencia recomendada: `libvips` vía `github.com/davidbyttow/govips/v2` (rápido, baja huella). Documentar instalación nativa requerida.
- **Por qué:** Aceptar imágenes raw del usuario es un riesgo de seguridad (ejecución vía polyglots, EXIF con geolocalización, archivos enormes que tumban el servicio) y de UX (servir originales pesados). Un pipeline obligatorio elimina toda esa clase de problemas.
- **Cuándo:** Bloqueante: `RDK-005`.

## Criterio de aceptación
- Paquete `shared/imaging` con pipeline registrable y configuración por *usage*.
- Tests:
  - Magic bytes inválidos → error tipado `ErrValidation`.
  - JPEG con EXIF GPS → output sin metadata (verificable con lectura de EXIF tras procesamiento).
  - Dimensiones fuera de rango → `ErrValidation`.
  - Cada variante respeta `MaxWidth`/`MaxHeight` y se persiste con la key esperada.
  - `HashSha256` corresponde al input recibido.
- Manejo de memoria: el pipeline no debe cargar archivos > umbral configurable en RAM.
- Documentado en `docs/imaging/pipeline.md` con ejemplo de declaración de un *usage* y de invocación.

## Fuera de alcance
- Detección automática de NSFW.
- Watermarking.
- AVIF (queda como mejora futura tras estabilizar WebP).
- OCR.

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que recibe uploads de imágenes desde usuarios.
- **Historia:** Como desarrollador que recibe uploads de imágenes, quiero un pipeline que valide, sanitice y genere variantes responsive automáticamente, para no exponer la aplicación a archivos maliciosos ni servir originales pesados al cliente.
- **Valor esperado:** Cualquier servicio que acepte imágenes lo hace con cero código propio de manejo de bytes; declara un *usage* y llama `Process`.
