# RDK-007 — Pipeline de PDFs seguros (validación, sanitización, persistencia privada).

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/pdf`
- **Épica técnica:** `EP-RDK-MEDIA` — Procesamiento de medios

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Entregar un pipeline para aceptar uploads de PDFs, validarlos, sanitizarlos (quitar JavaScript, AcroForms, attachments) y persistirlos con visibilidad privada por defecto.
- **Cómo:**
  - Pipeline:
    1. **Validación:** magic bytes `%PDF-`, tamaño máximo configurable (default 10MB), conteo de páginas máximo configurable.
    2. **Rechazar PDFs cifrados** (sin password popup en el cliente).
    3. **Sanitizar:** quitar `/JS`, `/JavaScript`, `/AA`, `/OpenAction`, AcroForms y attachments embebidos.
    4. **Persistir:** vía `StorageProvider` (RDK-005) con `Visibility = Private` por defecto. La key es `{namespace}/{usage}/{año}/{mes}/{uuid}.pdf`.
  - API:
    ```go
    func (p *PDFPipeline) Process(ctx context.Context, in io.Reader, usage string) (PDFPipelineResult, error)
    type PDFPipelineResult struct {
        Key        string
        SizeBytes  int64
        PageCount  int
        HashSha256 string
    }
    type PDFUsage struct {
        Name      string
        MaxBytes  int64
        MaxPages  int
        Visibility storage.Visibility
    }
    ```
  - Dependencia recomendada: `qpdf` vía syscall (más robusto y battle-tested) o `pdfcpu` Go puro como alternativa empaquetable. La elección debe ser configurable por driver.
  - Acceso al archivo persistido: signed URL del storage adapter con TTL corto (default 10 minutos).
- **Por qué:** Los PDFs son un vector clásico de ataque (JavaScript embebido, formularios que filtran datos al abrir, attachments con malware). Aceptar PDFs sin sanitizar pone en riesgo a quien los descargue.
- **Cuándo:** Bloqueante: `RDK-005`.

## Criterio de aceptación
- Paquete `shared/pdf` con pipeline registrable y configuración por *usage*.
- Tests:
  - PDF con `/JavaScript` → tras procesar no contiene la entrada (verificable con re-parsing).
  - PDF cifrado → `ErrValidation` con código `pdf.encrypted`.
  - PDF > tamaño máximo → `ErrValidation` con código `pdf.too_large`.
  - PDF con N páginas > `MaxPages` → `ErrValidation` con código `pdf.too_many_pages`.
  - PDF sanitizado se persiste como `Private` y se accede sólo con `SignedURL`.
- Documentado en `docs/pdf/pipeline.md` con ejemplo de declaración de *usage* e invocación.

## Fuera de alcance
- Thumbnails / previsualización de la primera página.
- OCR.
- Conversión a otros formatos.
- Watermarking.

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que recibe uploads de PDFs (reglamentos, bases, contratos, etc.).
- **Historia:** Como desarrollador que acepta PDFs subidos por usuarios, quiero un pipeline que rechace cifrados, sanitice JavaScript/AcroForms y persista en privado con URL firmada, para no exponer al resto de mis usuarios a un PDF malicioso.
- **Valor esperado:** Cualquier servicio que acepte PDFs lo hace sin tocar bytes manualmente; declara un *usage* y llama `Process`.
