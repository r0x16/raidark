# RDK-016 — Cliente HTTP inter-servicios estandarizado.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/httpx`
- **Épica técnica:** `EP-RDK-HTTP` — Capa HTTP base de Raidark

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Entregar un cliente HTTP común para llamadas síncronas entre servicios, que propague headers de auth y trazas, aplique timeouts, retries y mapee errores al envelope estándar (RDK-002).
- **Cómo:**
  - API:
    ```go
    type ServiceClient struct { /* ... */ }
    type ServiceClientOptions struct {
        Timeout         time.Duration
        Retries         int
        RetryBackoff    []time.Duration
        BaseHeaders     http.Header
    }
    func New(baseURL string, opts ServiceClientOptions) *ServiceClient
    func (c *ServiceClient) Do(ctx context.Context, req Request) (Response, error)

    type Request struct {
        Method  string
        Path    string
        Query   url.Values
        Headers http.Header
        Body    any            // serializa a JSON si no nil
    }
    type Response struct {
        Status  int
        Headers http.Header
        Body    []byte
    }
    func (r *Response) Decode(v any) error
    ```
  - Comportamiento:
    - Propaga automáticamente:
      - `Authorization` desde `ctx` (donde el middleware de auth del consumidor lo guardó).
      - `X-Correlation-ID` desde el `trace_id` en `ctx` (RDK-003 lo pobla).
      - `traceparent` W3C.
    - Timeout default por cliente; cancelable por `ctx`.
    - Retries con backoff configurable (default `[]time.Duration{200ms, 500ms, 1500ms}`):
      - Sólo reintenta sobre `5xx`, errores de red y timeouts; nunca sobre `4xx`.
    - Mapeo automático: si la respuesta tiene `Content-Type: application/json` y status >= 400, intenta parsear envelope `RESTError` (RDK-002) y devolverlo tipado.
  - Helpers convenientes:
    ```go
    func GetJSON[T any](ctx context.Context, c *ServiceClient, path string, q url.Values) (T, error)
    func PostJSON[T any](ctx context.Context, c *ServiceClient, path string, body any) (T, error)
    ```
- **Por qué:** Sin un cliente común, cada servicio reinventa retries, propagación de headers y mapeo de errores. La trazabilidad se rompe cuando alguien olvida propagar `traceparent`. Los retries inadecuados (sobre `4xx`) producen efectos duplicados.
- **Cuándo:** Bloqueante recomendado: `RDK-002` (para mapear errores), `RDK-003` (para propagar trace).

## Criterio de aceptación
- Tests con servidor HTTP de prueba:
  - Propagación de `Authorization` desde el contexto.
  - Propagación de `X-Correlation-ID` y `traceparent`.
  - Retry sobre `503` con backoff esperado, no retry sobre `400`.
  - Cancelación por `ctx` interrumpe el retry pendiente.
  - Decode de envelope `RESTError` cuando el servidor responde con shape estándar.
- Documentado en `docs/httpx/service-client.md` con ejemplo de configuración y uso.

## Fuera de alcance
- Service mesh / discovery dinámico (la baseURL viene por config).
- mTLS automático (si se requiere, lo configura el consumidor en el `http.Transport`).
- Rate limiting client-side (otra tarea futura).

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que llama a otros servicios sincrónicamente desde un servicio sobre Raidark.
- **Historia:** Como desarrollador que hace llamadas inter-servicios, quiero un cliente HTTP estándar que propague auth y trazas y maneje retries inteligentes, para no perder correlación entre servicios y para no duplicar requests por reintentos mal configurados.
- **Valor esperado:** Cualquier llamada inter-servicios queda observable end-to-end y robusta ante fallas transitorias sin código por servicio.
