# RDK-014 — Sanitizador markdown.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/sanitizer/markdown`
- **Épica técnica:** `EP-RDK-CONTENT` — Sanitización de contenido

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Proveer un sanitizador para texto markdown subido por usuarios, que rinda HTML seguro y devuelva también el markdown limpio (sin construcciones peligrosas).
- **Cómo:**
  - API:
    ```go
    type Policy int
    const (
        PolicyDefault Policy = iota
        PolicyStrict           // chat-like: sin imágenes, sin links a dominios externos
        PolicyArticle          // posts/news/events: imágenes locales sí, scripts no, iframes no
    )
    func Sanitize(input string, policy Policy) (cleanMarkdown string, safeHTML string, err error)
    ```
  - Implementación:
    - Parser markdown: `github.com/gomarkdown/markdown` o `goldmark`.
    - Sanitizador HTML: `github.com/microcosm-cc/bluemonday` con `UGCPolicy` extendida:
      - Sin `<iframe>`.
      - Sin `<script>`.
      - Sin atributos `style`, `on*`.
      - Sin imágenes externas en `PolicyStrict`.
      - Allowlist de protocolos: `http`, `https`, `mailto`. Sin `javascript:` ni `data:` salvo casos puntuales declarados por la política.
    - Política `PolicyArticle` admite imágenes con dominios listados (configurables en init de la policy).
  - Permitir extender políticas: el consumidor puede registrar una policy custom con `RegisterPolicy(name string, p *bluemonday.Policy)` y usarla por nombre.
- **Por qué:** Persistir HTML producido por el cliente o markdown sin sanitizar es XSS estructural. Un sanitizador centralizado garantiza una sola implementación, una sola superficie de bugs y una sola política revisable.
- **Cuándo:** Sin bloqueantes.

## Criterio de aceptación
- Tests:
  - Input con `<script>` → no aparece en `safeHTML`.
  - Input con `<iframe src=...>` → no aparece en `safeHTML`.
  - Input con `[link](javascript:alert(1))` → link removido o protocolo neutralizado.
  - Input con `<img src="https://evil.example/...">` bajo `PolicyStrict` → imagen removida.
  - Mismo input bajo `PolicyArticle` con dominio en allowlist → imagen presente.
  - `cleanMarkdown` round-trip: al re-sanitizar, no cambia.
- Documentado en `docs/sanitizer/markdown.md` con descripción de cada policy y guía para registrar policies custom.

## Fuera de alcance
- Sanitización de markdown extendida a otros formatos (BBCode, AsciiDoc).
- Renderer rico (con sintaxis highlighting, mermaid, etc.).
- Detección de spam / contenido tóxico (otra tarea, no es sanitización).

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que persiste contenido markdown subido por usuarios.
- **Historia:** Como desarrollador que persiste markdown user-generated, quiero un sanitizador con políticas configurables, para garantizar que no estoy guardando XSS y para que la política de un canal (chat) sea más estricta que la de un artículo editorial.
- **Valor esperado:** Una sola implementación de sanitización en todo el ecosistema. Cambiar política es cambiar un parámetro.
