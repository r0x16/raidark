# RDK-015 — Idempotency-Key store.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/api/idempotency`
- **Épica técnica:** `EP-RDK-HTTP` — Capa HTTP base de Raidark

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Entregar un store y un wrapper de handler para soportar el header `Idempotency-Key` en endpoints `POST` que crean recursos o disparan side-effects no idempotentes.
- **Cómo:**
  - Tabla `idempotency_keys` (modelo GORM exportable):
    | Columna | Tipo | Notas |
    |---|---|---|
    | `key` | text | PK lógica junto a `endpoint` y `user_id` |
    | `endpoint` | text | |
    | `user_id` | text | |
    | `request_hash` | text | hash del body para detectar reuso del key con body distinto |
    | `response_status` | int | |
    | `response_body` | bytea | |
    | `created_at` | timestamptz | default now() |
    | `expires_at` | timestamptz | TTL configurable, default 24h |
    PK compuesta: `(user_id, endpoint, key)`.
  - Helper:
    ```go
    func WithIdempotency(c echo.Context, fn func() (status int, body any, err error)) error
    ```
    - Lee `Idempotency-Key` header. Si está vacío y la política lo exige, retorna 400 (`idempotency.required`).
    - Lookup por `(user_id, endpoint, key)`:
      - No existe → ejecuta `fn`, persiste status+body, devuelve respuesta.
      - Existe con `request_hash` igual → devuelve la respuesta cacheada.
      - Existe con `request_hash` distinto → 409 `idempotency.mismatch`.
    - Limpieza basada en `expires_at` la opera infraestructura (job aparte) o el consumidor.
  - `user_id` se extrae del contexto de auth (consumidor del framework registra un `UserIDExtractor` similar al `PermissionExtractor` de RDK-013).
  - **Soporte multi-engine GORM** (los tres drivers que Raidark ya soporta):
    - **Postgres**: implementación primaria. `SELECT ... FOR UPDATE` garantiza serialización entre requests concurrentes con la misma key. Caso recomendado para producción.
    - **MySQL**: usa `SELECT ... FOR UPDATE` también; comportamiento equivalente a Postgres.
    - **SQLite**: no soporta `FOR UPDATE`. La implementación cae a una transacción `IMMEDIATE` (escritura inmediata bloquea la base) + retry breve. Limitación: throughput menor y posible `SQLITE_BUSY` bajo concurrencia alta. Documentar en `docs/api/idempotency.md` que SQLite es para desarrollo y tests, no producción.
    - El driver concreto se detecta automáticamente desde el `*gorm.DB`. No hay configuración separada.
- **Por qué:** Sin un store común, cada endpoint que acepte `Idempotency-Key` implementa su propia tabla y lógica, con riesgo de divergencia. Un único helper garantiza comportamiento uniforme y documentable.
- **Cuándo:** Bloqueante recomendado: `RDK-002` (envelope de error) para responder 409 con shape estándar.

## Criterio de aceptación
- Modelo `IdempotencyKey` exportable como hook de migración.
- Helper `WithIdempotency` exportado, agnóstico al driver de DB.
- Tests con Postgres testcontainer **y** MySQL testcontainer **y** SQLite in-memory:
  - Primera invocación con un key → ejecuta `fn` y persiste.
  - Segunda invocación misma key + mismo body → devuelve respuesta cacheada sin re-ejecutar `fn`.
  - Segunda invocación misma key + body distinto → 409 `idempotency.mismatch`.
  - Key expirado → se trata como nuevo (re-ejecuta).
  - Concurrencia (sólo Postgres y MySQL): dos requests con misma key llegan en paralelo; sólo una ejecuta `fn`, la otra recibe la cacheada.
  - SQLite: documentar y testear el comportamiento bajo `SQLITE_BUSY` (retry o error claro).
- Documentado en `docs/api/idempotency.md` con sección "Engine support" explicando garantías y limitaciones por motor.

## Fuera de alcance
- Limpieza automática de expirados (responsabilidad de un job operativo aparte).
- Garantías fuertes de concurrencia en SQLite (no es soportado por el motor; queda como uso recomendado sólo para desarrollo y tests).

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que expone endpoints `POST` no idempotentes.
- **Historia:** Como desarrollador que expone un endpoint POST que crea recursos o dispara side-effects, quiero un store de Idempotency-Key reutilizable, para que un cliente que reintenta tras un timeout no me cree el recurso dos veces.
- **Valor esperado:** Cualquier handler puede aceptar `Idempotency-Key` envolviéndose con `WithIdempotency`, sin escribir lógica propia.
