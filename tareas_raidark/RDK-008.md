# RDK-008 — Envelope estándar de evento de dominio.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/events/domain`
- **Épica técnica:** `EP-RDK-EVENTS` — Bus de eventos de dominio

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Definir un envelope versionado y trazable para todo evento de dominio que viaje por el bus, sin obligar al `DomainEvent` actual (hoy expone `Name()` y `OccurredAt()`) a conocer infraestructura.
- **Cómo:**
  - Tipo Go `EventEnvelope` con campos:
    | Campo | Tipo | Descripción |
    |---|---|---|
    | `event_id` | string (UUIDv7) | Identificador único del mensaje. Permite dedup. |
    | `event_name` | string | Corresponde a `DomainEvent.Name()`. |
    | `event_version` | int | Versión del schema del payload. Empieza en `1`. |
    | `occurred_at` | RFC3339Nano UTC | Hora lógica del dominio (no de publicación). |
    | `published_at` | RFC3339Nano UTC | Hora de publicación por el driver. |
    | `producer` | string | Servicio emisor. |
    | `trace_id` | string | W3C trace-context propagado desde HTTP/consumer. |
    | `span_id` | string | W3C span-context del productor. |
    | `idempotency_key` | string | Default = `event_id`. Sobrescribible al reemitir. |
    | `correlation_id` | string (opcional) | Agrupa eventos de una misma intención. |
    | `payload` | any | JSON del evento de dominio concreto. |
  - Helper:
    ```go
    func NewEnvelope(event DomainEvent, producer string, traceCtx context.Context) EventEnvelope
    ```
    - Genera `event_id` con `ids.NewV7()` (RDK-001).
    - Pobla `trace_id` y `span_id` desde `traceCtx`.
    - `published_at` queda vacío (lo setea el publisher al enviar).
    - `idempotency_key` default `= event_id`.
  - Serialización JSON con shape estable. La convención (camelCase vs snake_case) se elige una sola vez y se documenta. Recomendado: snake_case para alinearse con headers HTTP y subjects.
  - Reglas de versionado del payload (documentadas en `docs/events/envelope.md`):
    - Cambios compatibles hacia atrás → no suben `event_version`.
    - Cambios incompatibles → suben `event_version` y se publica en otro subject (responsabilidad del consumidor del envelope, no de Raidark).
    - Consumidor ignora campos desconocidos; productor nunca remueve campos dentro de la misma versión.
- **Por qué:** Sin envelope versionado y trazable, cualquier cambio de schema rompe consumidores y cualquier bug distribuido es indebuggeable. El envelope desacopla infraestructura de dominio: el `DomainEvent` no necesita conocer trace, idempotencia ni versionado.
- **Cuándo:** Bloqueante: `RDK-001`.

## Criterio de aceptación
- Tipo `EventEnvelope` y helper `NewEnvelope` exportados.
- Tests unitarios:
  - Round-trip JSON estable (mismo bytes en serial / deserial / reserial).
  - Default `idempotency_key = event_id`.
  - `trace_id` y `span_id` extraídos del contexto cuando están presentes.
  - `published_at` vacío en construcción, asignable después.
- Documentado en `docs/events/envelope.md` con esquema, convenciones, ejemplo y reglas de versionado.

## Fuera de alcance
- Publisher y consumer (ver RDK-009 / RDK-010).
- Naming de subjects (responsabilidad del consumidor del framework; Raidark no impone convención).

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que emite o consume eventos de dominio.
- **Historia:** Como desarrollador que emite o consume eventos de dominio sobre Raidark, quiero un envelope estándar versionado y trazable, para correlacionar eventos con sus orígenes HTTP, evolucionar schemas sin romper consumidores y deduplicar a nivel de mensaje.
- **Valor esperado:** Todos los eventos comparten una misma forma exterior; las herramientas de tracing y dedup funcionan sin código específico por servicio.
