# RDK-018 — EmailSender abstracto con drivers SMTP y HTTP.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/email`
- **Épica técnica:** `EP-RDK-NOTIFICATIONS` — Notificaciones salientes

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Definir una interfaz `EmailSender` con dos drivers iniciales (SMTP y HTTP) para soportar tanto servidores propios como providers tipo SES / Postmark / SendGrid.
- **Cómo:**
  - API:
    ```go
    type EmailSender interface {
        Send(ctx context.Context, msg EmailMessage) error
    }
    type EmailMessage struct {
        To       string
        Subject  string
        BodyText string
        BodyHTML string
        From     string
        Headers  map[string]string
    }
    ```
  - Driver `smtp`: vía `net/smtp` o `github.com/wneessen/go-mail`. Soporta TLS implícito y STARTTLS.
  - Driver `http_generic`: cliente sobre `RDK-016` que envía a un endpoint configurable con un payload JSON arbitrario (mapping de campos `EmailMessage` → cuerpo del POST configurable por env). Útil para integrar cualquier provider HTTP sin tocar Raidark.
  - Driver `http_gws` (Google Workspace Gmail API): envía vía `users.messages.send` de la Gmail API usando una cuenta de servicio con domain-wide delegation. Soporta enviar como cualquier usuario del workspace.
  - Driver `http_brevo` (Brevo, ex Sendinblue): envía vía `https://api.brevo.com/v3/smtp/email`. Brevo ofrece **300 correos diarios gratis sin vencimiento** (~9k/mes), lo que lo hace una opción razonable de bootstrap sin costo recurrente.
  - Manejo de errores:
    - `4xx` del provider o respuesta SMTP `5xx` (rejected) → `ErrPermanent`.
    - `5xx` del provider o `4xx` SMTP (try later) → `ErrTransient`.
    - `429` (rate limit) → `ErrTransient` con backoff respetando `Retry-After` si viene.
  - Provider factory `EmailSenderFactory` registrable. Cada driver entrega su propia factory (`SMTPSenderFactory`, `HTTPGenericSenderFactory`, `GWSSenderFactory`, `BrevoSenderFactory`).
- **Por qué:** Mandar correo es transversal y propenso a integraciones específicas. Una abstracción permite que el código de aplicación arme el `EmailMessage` y delegue el transporte, sin acoplarse a un proveedor.
- **Cuándo:** Bloqueante recomendado: `RDK-016` para el driver HTTP.

## Variables de entorno
```
EMAIL_DRIVER=smtp|http_generic|http_gws|http_brevo
EMAIL_DEFAULT_FROM="Notifications <no-reply@example.com>"

# smtp
SMTP_HOST=...
SMTP_PORT=587
SMTP_USERNAME=...
SMTP_PASSWORD=...
SMTP_TLS=starttls|implicit|none

# http_generic
EMAIL_HTTP_ENDPOINT=...
EMAIL_HTTP_AUTH_HEADER=Authorization
EMAIL_HTTP_AUTH_TOKEN=...
EMAIL_HTTP_PAYLOAD_TEMPLATE=<json template con placeholders {{.To}}, {{.From}}, {{.Subject}}, {{.BodyText}}, {{.BodyHTML}}>

# http_gws (Google Workspace Gmail API)
GWS_SERVICE_ACCOUNT_JSON=/secrets/gws-sa.json
GWS_IMPERSONATE_USER=no-reply@example.com

# http_brevo
BREVO_API_KEY=xkeysib-...
```

## Criterio de aceptación
- Interfaz y los cuatro drivers exportados (`smtp`, `http_generic`, `http_gws`, `http_brevo`).
- Tests:
  - Driver SMTP contra mailpit / mailhog testcontainer; mensaje recibido con headers y body correctos.
  - Driver `http_generic` contra mock server: payload renderiza correctamente el `EMAIL_HTTP_PAYLOAD_TEMPLATE`.
  - Driver `http_gws` contra mock de Gmail API: payload base64url-encoded del MIME, header `Authorization: Bearer <token>` con token obtenido vía service account.
  - Driver `http_brevo` contra mock de Brevo: payload JSON con shape `{ sender, to, subject, htmlContent, textContent }`, header `api-key`.
  - Clasificación de errores: `5xx` del HTTP provider → `ErrTransient`, `400/401/403` con body de error → `ErrPermanent`, `429` → `ErrTransient` respetando `Retry-After`.
- Documentado en `docs/email/sender.md` con ejemplos de configuración por driver, incluyendo cómo crear la service account de GWS y cómo obtener la API key de Brevo.

## Fuera de alcance
- Templating (responsabilidad del consumidor; p. ej. `text/template` o un sistema MJML aparte).
- Tracking de aperturas / clicks.
- Bounces y suppressions automáticos (los gestiona el provider; consumidor decide cómo reaccionar).

## Historia de usuario relacionada
- **Actor:** Equipo desarrollador que envía correos transaccionales desde un servicio.
- **Historia:** Como desarrollador que manda correos transaccionales, quiero una interfaz `EmailSender` con drivers SMTP y HTTP, para empezar con un servidor SMTP simple y migrar a un provider sin reescribir mi código.
- **Valor esperado:** El código de aplicación arma `EmailMessage` y llama `Send`. Cambiar driver es cambiar configuración.
