# RDK-003 — Observabilidad base: logs estructurados, métricas Prometheus y W3C trace context.

## Ubicación
- **Repositorio:** Raidark (`github.com/r0x16/Raidark`)
- **Componente:** `shared/observability`
- **Épica técnica:** `EP-RDK-OBSERVABILITY` — Observabilidad transversal

## Tarea técnica
- **Tipo:** DEVELOPMENT
- **Estado:** Ready
- **Quién:** DEV
- **Qué:** Entregar la observabilidad base que cualquier servicio construido sobre Raidark necesita: logs estructurados con correlación, métricas Prometheus mínimas y propagación W3C trace context entre HTTP y bus de eventos.
- **Cómo:**
  - Extender `LogProvider` actual con un adapter JSON sobre `slog` (Go 1.21+) o `zap`. Toggleable por env (`LOG_FORMAT=json|text`).
  - Helper `log.FromContext(ctx)` que devuelve un logger con campos automáticos: `trace_id`, `span_id`, `service`, y `event_id` cuando aplique.
  - Middleware Echo `observability.HTTPMetrics()` que expone `http_requests_total{status,endpoint}` y `http_request_duration_ms{endpoint}` (histograma con buckets razonables: 5, 25, 100, 500, 1000, 5000 ms).
  - Counter/gauge/histogram registrados para eventos: `events_published_total{subject,outcome}`, `events_consumed_total{subject,consumer,outcome}`, `events_redeliveries_total{subject,consumer}`, `event_processing_duration_ms{subject,consumer}`, `outbox_pending_gauge`.
  - Middleware `observability.W3CTrace()` que parsea `traceparent` / `tracestate` entrantes o genera uno nuevo, pueblan `trace_id` y `span_id` en `echo.Context`.
  - Provider factory `PrometheusMetricsProvider` que monta `/metrics` en el router cuando `METRICS_ENABLED=true`.
  - Helper para drivers NATS/HTTP de propagar `traceparent` en headers (consumido por RDK-009/010 y RDK-016).
- **Por qué:** Sin observabilidad mínima estandarizada, cada servicio reinventa logging y métricas con campos distintos, y debuggear cross-service en presencia de eventos asíncronos se vuelve impracticable.
- **Cuándo:** Bloqueante recomendado: `RDK-002` (para reusar `X-Correlation-ID` como `trace_id` cuando ya viene poblado).

## Variables de entorno
```
LOG_FORMAT=json
LOG_LEVEL=INFO
METRICS_ENABLED=true
METRICS_PATH=/metrics
SERVICE_NAME=<nombre del servicio>
```

## Criterio de aceptación
- Paquete `shared/observability` con los middlewares y los métricas registrados.
- Logger automático inyecta `trace_id`, `span_id`, `service`.
- `/metrics` expone counters/histograms con labels documentados.
- Tests:
  - Log emitido contiene `trace_id` cuando el contexto lo trae.
  - Histogram de HTTP recibe muestras tras una request real (test con Echo).
  - W3C: `traceparent` válido entrante se propaga; ausente genera uno nuevo.
- Documentado en `docs/observability/logging.md`, `docs/observability/metrics.md`, `docs/observability/tracing.md`.

## Fuera de alcance
- OpenTelemetry collector y exporters (OTLP). Se deja la propagación W3C lista para que el siguiente paso sea integrar OTel sin tocar handlers.
- Dashboards Grafana.
- Alerting.

## Historia de usuario relacionada
- **Actor:** SRE / on-call que opera servicios construidos sobre Raidark.
- **Historia:** Como SRE que opera servicios construidos sobre Raidark, quiero logs JSON con `trace_id` automático, métricas Prometheus en `/metrics` y propagación W3C trace, para correlacionar requests entre servicios y entre HTTP y eventos sin instrumentar cada servicio a mano.
- **Valor esperado:** Cada servicio nuevo expone observabilidad base sin código adicional. Una traza HTTP que dispara un evento sigue siendo correlacionable cuando el evento es consumido en otro proceso.
